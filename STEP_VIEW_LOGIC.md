# Step View 튜토리얼 생성 로직

## 1. 전체 흐름
1. Step View가 열리면 `DECISIONS.md`와 `.ai-tutorials/*.md`를 감시하는 watcher를 등록하고, 현재 로그 엔트리와 저장된 튜토리얼 목록을 읽어 webview 상태로 보낸다.
2. webview는 받은 엔트리를 체크박스 목록으로 렌더링하고, 선택 상태를 `selectedIds` 집합으로 유지한다. `전체 선택`, `선택 해제`, `문서 생성` 버튼은 이 집합 크기에 따라 활성화된다.
3. 사용자가 `문서 생성`을 누르면 현재 선택된 step id 배열을 `generateTutorial` 메시지로 확장 쪽에 보낸다.
4. `StepViewController.handleGenerateTutorial()`는 먼저 선택 id가 비어 있지 않은지 확인하고, 다시 `DECISIONS.md` 전체를 읽은 뒤 id 기준으로 실제 엔트리를 찾아 선택 목록을 재구성한다.
5. 선택된 엔트리가 하나도 없으면 즉시 오류를 반환하고 끝낸다. 선택된 엔트리가 있으면 `AIClient.generateTutorial(selectedEntries)`를 호출해 LLM에 markdown 생성을 요청한다.
6. LLM이 반환한 문자열은 후처리 없이 그대로 markdown 본문으로 간주된다. 이 시점에는 형식 검증이나 섹션 누락 확인이 없다.
7. 제목은 선택 step이 1개면 해당 step 제목 기반으로, 여러 개면 첫 번째 step 제목과 개수 기반으로 만든 뒤 파일명에 안전한 문자만 남기도록 정리한다.
8. `LogManager.saveTutorial()`가 워크스페이스 아래 `.ai-tutorials` 디렉터리를 만들고, 생성된 markdown을 `.md` 파일로 저장한다.
9. 저장 직후 Step View는 상태를 다시 새로고침하고, 생성된 파일을 VS Code 편집기로 바로 연다.
10. webview는 `tutorialGenerated` 메시지를 받으면 선택 상태를 비우고, 몇 개 step으로 문서를 만들었는지 상태 배너로 보여 준다.

## 2. DECISIONS.md 파싱 방식
`LogManager.readLogEntries()`는 먼저 `DECISIONS.md`가 없으면 빈 파일을 만든 다음 전체 텍스트를 읽는다. 파싱은 `## Step:`로 시작하는 블록 단위 분할 방식이며, 각 블록에서 제목, 날짜, 질문, Option A, Option B, 사용자 선택, 결과를 정규식으로 추출한다. 각 엔트리 객체는 `id`, `title`, `date`, `question`, `optionA`, `optionB`, `userChoice`, `outcome` 필드를 가진다. `id`는 `title::date`를 base64url로 인코딩해 만든다. 최종 배열은 파일 안 기록 순서를 뒤집어 가장 최근 step이 먼저 오도록 반환한다.

## 3. generateTutorial() 입력 구조
`generateTutorial()`의 입력은 사용자가 선택한 `DecisionLogEntry[]` 배열이다. 여기에는 step 제목, 날짜, 질문, 두 선택지 요약, 사용자 최종 선택, 결과가 모두 포함된다. 함수는 이 배열을 `## Entry 1`, `## Entry 2` 같은 블록으로 펼치고, 각 블록마다 제목, 날짜, 질문, Option A, Option B, 사용자 선택, 결과를 순서대로 넣어 단일 user prompt 문자열로 만든다. 별도 워크스페이스 컨텍스트, AGENT.md, 코드 스냅샷은 붙지 않고 선택된 decision log 정보만 전달된다. 모델 호출은 `sendMessage(STEP_SYSTEM_PROMPT, userPrompt, 1800)` 형태로 실행된다.

## 4. STEP_SYSTEM_PROMPT 현재 구조
프롬프트는 현재 깨진 한글이 많지만 구조상으로는 먼저 “개발자의 판단 로그를 학습용 문서로 재구성하는 역할”을 지정한다. 다음으로 전체 문서를 한국어로 쓰고 단순 요약이 아니라 트레이드오프가 드러나는 문서로 만들라는 지시가 들어간다. 그 뒤 문서 상단에 프로젝트 제목과 프로젝트 맥락을 두라고 요구한다. 각 선택 step마다 `결정한 것`, `왜 필요했나`, `선택지 비교표`, `이 프로젝트에서 선택한 이유`, `이 선택이 이후 결정에 미친 영향`, `이 판단이 틀렸을 때 나타날 신호`, `다음에 비슷한 상황이 오면`을 넣으라고 지시한다. 비교표는 markdown 표로, 본문은 bullet보다 prose 문단으로 쓰라고 요구한다. 여러 step을 함께 다룰 때는 마지막에 `판단들의 연결 구조`와 `내 판단 패턴 분석`을 추가하라고 요구한다. 마지막으로 교과서보다 시니어 개발자의 복기 문체를 유지하라는 톤 지시가 붙어 있다.

## 5. 출력 포맷 강제 방식
출력 포맷 강제는 전적으로 system prompt 문장에 의존한다. 모델 응답은 markdown 문자열로 그대로 받아 저장되며, section 존재 여부를 검사하는 파서나 validator는 없다. LLM이 특정 섹션을 생략하거나 표 형식을 어겨도 이를 보정하는 후처리는 없다. 응답은 JSON이 아니라 자유 텍스트이기 때문에 `parseJsonResponse()` 같은 구조 검증도 적용되지 않는다. 결과적으로 현재 포맷 강제는 “프롬프트 지시” 수준이고, 실행 단계의 보장 장치는 없는 상태다.

## 6. 현재 한계
- `STEP_SYSTEM_PROMPT`와 `generateTutorial()`의 입력 문구에 한글 깨짐이 남아 있어, 의도한 학습 문서 구조가 모델에 안정적으로 전달되지 않을 수 있다.
- 튜토리얼 생성에는 선택된 decision log만 들어가고 실제 코드, 워크스페이스 구조, AGENT.md, 구현 결과 요약은 포함되지 않아 맥락 폭이 좁다.
- 출력 길이 예산이 `1800`으로 고정돼 있어 여러 step을 동시에 선택하면 step당 설명 밀도가 급격히 낮아질 수 있다.
- 출력 포맷 검증이 전혀 없어서 섹션 누락, 표 누락, 문체 붕괴가 발생해도 저장 단계에서 그대로 통과한다.
- 생성된 markdown에 대해 품질 점수, 최소 섹션 수, 길이 제한, 연결 구조 존재 여부를 확인하는 후처리도 없다.
- 제목 생성은 첫 번째 step 제목과 개수에 의존하므로, 서로 다른 주제 step을 여러 개 묶었을 때 파일명이 내용 전체를 잘 대표하지 못할 수 있다.
- Step View UI와 오류 메시지 일부가 아직 깨져 있어 실제 사용자는 튜토리얼 생성 상태를 일관되게 이해하기 어려울 수 있다.
- `DECISIONS.md` 파싱이 완전히 텍스트 패턴 기반이라 로그 포맷이 조금만 달라져도 튜토리얼 생성 입력 자체가 깨질 수 있다.
