# PATCH 04 - Workspace Context Relevance Filter

## 추가된 랭킹 로직

- 사용자 요청 문자열에서 공백 기준으로 키워드를 분리합니다.
- `은 / 는 / 이 / 가 / 을 / 를 / 에 / 의 / 로 / 으로 / and / the / a` 같은 불용어는 제거합니다.
- 각 파일은 아래 두 위치에서 키워드가 몇 개나 등장하는지로 점수를 계산합니다.
  - 파일 경로와 파일명
  - 파일 내용의 앞 50줄
- 점수는 `경로/이름 매칭 수 + 앞 50줄 매칭 수`로 계산합니다.
- `preferredPaths`가 들어온 경우에는 그 경로와 정확히 일치하거나 하위/상위 관계인 파일을 우선순위 상단으로 올립니다.
- `package.json`, `tsconfig.json`, `go.mod`, `Cargo.toml`, `requirements.txt`는 점수가 0이어도 항상 포함 대상으로 취급합니다.
- 최종적으로는 점수 높은 파일부터 정렬하고, 문자 예산 안에서만 순서대로 snapshot에 넣습니다.

## 항상 제외되는 파일

- `node_modules/**`
- `.git/**`
- `.vendor/**`
- `dist/**`
- `build/**`
- `out/**`
- `.ai-tutorials/**`
- `DECISIONS.md`
- `AGENT.md`
- `*.lock`
- `*.map`
- 기존처럼 이미지/압축/PDF 같은 명백한 바이너리 자원도 계속 제외합니다.

## 문자 예산 제한

- snapshot 전체 문자 수 상한은 `8000자`입니다.
- 헤더를 먼저 넣고, 그 다음 관련 파일 섹션을 순서대로 추가합니다.
- 각 파일 섹션은 `FILE: 경로 + 앞부분 미리보기` 형태로 들어가며, 다음 섹션을 넣었을 때 8000자를 넘으면 잘라내거나 제외합니다.

## 왜 8000자로 제한했는가

- planning 프롬프트에 불필요하게 긴 워크스페이스 덤프가 들어가면 토큰만 늘고 관련성은 오히려 떨어집니다.
- 8000자 제한은 "관련 파일 몇 개의 핵심 단서"만 모델에 주고, 나머지 노이즈를 잘라서 planning 품질과 비용 효율을 같이 잡기 위한 제한입니다.
