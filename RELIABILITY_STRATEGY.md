# Debtcrasher 신뢰성 우려 대응 전략

## 1. 문서 목적

이 문서는 Debtcrasher를 논문, 발표, 지도교수 피드백 상황에서 설명할 때 예상되는 신뢰성 우려에 대응하기 위한 논리와 최소 구현 대책을 정리한다.

핵심 목표는 다음과 같다.

- 기존에 널리 사용되는 오픈소스 AI 개발 에이전트 래퍼를 변형해 사용했다는 점을 신뢰성 방어 논리로 정리한다.
- 고레버리지 질문을 판단하는 기준을 명시해 "왜 이 질문을 사용자에게 물어보는가"를 설명 가능하게 만든다.
- 학습 자료 생성 과정에서 발생할 수 있는 환각, 왜곡, 과잉 일반화 문제를 낮은 비용으로 줄이는 방안을 제시한다.
- 개발자를 대상으로 하는 도구라는 특성상 과도한 검증으로 사용성을 해치지 않는다는 설계 철학을 논문용 표현으로 바꾼다.

## 2. 핵심 주장 요약

Debtcrasher의 신뢰성 전략은 "모든 AI 출력을 완전히 검증한다"가 아니다. 이 프로젝트의 대상 사용자는 개발자이며, 개발자는 생성 결과를 그대로 신뢰하는 최종 소비자가 아니라 코드, 테스트, diff, 실행 결과를 검토할 수 있는 실무 사용자다.

따라서 Debtcrasher는 고비용 검증을 기본값으로 두기보다, 생산성을 해치지 않는 범위에서 다음 세 가지 지점에만 신뢰성 장치를 둔다.

1. 검증된 오픈소스 기반 구조를 재사용해 도구 자체의 엔지니어링 리스크를 낮춘다.
2. 중요한 의사결정이 필요한 경우에만 고레버리지 질문을 생성한다.
3. 생성된 학습 자료가 실제 결정 로그와 코드 변경에 근거하도록 최소한의 근거 추적 장치를 둔다.

논문에서는 이를 "신뢰성을 포기한 설계"가 아니라 "개발자 생산성과 검증 비용 사이의 균형을 고려한 friction-aware reliability design"으로 설명하는 것이 적절하다.

## 3. 오픈소스 기반 사용을 통한 신뢰성 방어

### 3.1 소스코드 기반 추정: Continue 기반 변형 가능성이 가장 높음

현재 저장소의 소스코드와 메타데이터를 기준으로 보면, Debtcrasher가 가장 직접적으로 기반을 둔 오픈소스는 **Continue**로 추정된다. 단순한 외형 유사성 때문이 아니라, 코드 내부에 Continue를 명시한 주석과 NOTICE가 남아 있기 때문에 이 추정의 신뢰도는 높다.

추정 근거는 다음과 같다.

- `.vendor/continue/` 디렉터리가 존재하며 Continue 원본 저장소 구조가 함께 보관되어 있다.
- `NOTICE` 파일에 "architecture and implementation patterns adapted from Continue"라고 명시되어 있다.
- `src/activation/activateDebtcrasher.ts`에는 "Activation flow adapted from Continue's activation bootstrap" 주석이 있다.
- `src/extension/DebtcrasherExtension.ts`에는 "Architectural structure adapted from Continue's VsCodeExtension orchestration" 주석이 있다.
- `src/context/WorkspaceContextService.ts`에는 "Architectural structure adapted from Continue's VS Code extension layering" 주석이 있다.
- `package.json`의 라이선스가 Apache-2.0이고, Continue도 Apache-2.0 기반이어서 라이선스 계열이 맞다.
- 구조적으로도 VS Code activity bar container, webview view provider, extension orchestrator, workspace context 수집, provider 설정 흐름이 Continue 계열의 VS Code 확장 구조와 가장 가깝다.

따라서 발표와 논문에서는 다음과 같이 말하는 것이 적절하다.

```text
Debtcrasher는 Continue의 VS Code 확장 아키텍처와 실행 흐름을 참고해 개발한 변형 프로젝트로 볼 수 있다.
다만 Continue의 일반 코딩 어시스턴트 기능 전체를 복제한 것이 아니라, 그 위에 planning gate,
decision log, step-based tutorial generation을 결합해 개발자의 의사결정 학습을 지원하도록 재구성했다.
```

### 3.2 Continue를 기반으로 보아야 하는 이유

Continue는 VS Code/JetBrains 계열 IDE에서 동작하는 오픈소스 AI 개발 도구이며, 여러 모델 provider와 개발자 워크플로를 연결하는 구조를 갖고 있다. 2026-04-28 GitHub 확인 기준으로 `continuedev/continue` 저장소는 약 32.8k stars, 4.4k forks, 21,498 commits, Apache-2.0 license, 822 releases를 보유하고 있었다.

이 수치는 "Continue가 오류 없이 완벽하다"는 근거가 아니라, 다음 정도의 제한된 신뢰성 근거로 사용하는 것이 안전하다.

- 많은 개발자가 관심을 갖고 사용하는 공개 프로젝트다.
- 유지보수와 릴리스 이력이 있는 프로젝트다.
- Apache-2.0 라이선스 기반이라 변형과 재사용의 법적 근거를 설명하기 쉽다.
- VS Code 확장, webview, provider 설정 같은 기반 구조를 참고했다는 점을 방어할 수 있다.

반대로 이 수치만으로 다음을 주장하면 안 된다.

- Debtcrasher의 AI 출력이 정확하다는 주장
- 학습 자료 생성이 항상 사실이라는 주장
- Continue 기반이므로 별도 검증이 필요 없다는 주장

가장 안전한 표현은 다음과 같다.

```text
본 프로젝트는 널리 사용되는 오픈소스 AI 개발 도구인 Continue의 VS Code 확장 구조를 참고하여
도구 인프라 계층의 구현 리스크를 낮추고, 연구 기여를 planning gate와 decision-grounded
learning note generation에 집중하였다.
```

### 3.3 다른 후보와의 구분

소스코드만 기준으로 보면 Cline, Roo Code, Cody, OpenHands를 직접 기반으로 삼았다는 근거는 약하다.

| 후보 | 소스코드 기준 판단 | 이유 |
| --- | --- | --- |
| Continue | 직접 기반 가능성 높음 | `.vendor/continue`, NOTICE, 여러 source comment에서 명시 |
| Cline | UI 아이디어 참고 가능성은 있으나 직접 기반 근거 약함 | 코드 주석이나 vendor 디렉터리에서 직접 흔적 없음 |
| Roo Code | 직접 기반 근거 약함 | mode 분리 아이디어와 유사할 수 있으나 코드 흔적 없음 |
| Cody | 직접 기반 근거 약함 | 대규모 context/IDE assistant라는 일반적 유사성만 있음 |
| OpenHands | 직접 기반 근거 약함 | 범용 agent loop와 목적이 다르고 VS Code 확장 기반 흔적 없음 |

따라서 논문에서는 "Continue 기반 구조를 변형했고, Cline류 agent의 phase UI는 참고 가능한 관련 사례로 언급할 수 있다" 정도가 가장 방어 가능하다.

정리하면 다음과 같다.

```text
Base architecture: Continue
Possible UI inspiration: Cline-style task phase visualization
Debtcrasher contribution: high-leverage planning gate, persistent decision memory, step-based learning material generation
```

### 3.4 주장 방식

강하게 주장하면 안 되는 표현:

- "많은 개발자가 문제 없이 사용했으므로 안전하다."
- "오픈소스 기반이므로 신뢰성 문제가 없다."
- "기존 래퍼를 썼기 때문에 검증이 필요 없다."

논문과 발표에서 사용하기 좋은 표현:

- "Debtcrasher는 AI 개발 에이전트 래퍼의 기본 실행 구조를 새로 발명하지 않고, 이미 공개적으로 사용되고 유지보수되는 오픈소스 구조를 기반으로 변형하였다."
- "이를 통해 VS Code 확장 생명주기, webview 통신, 모델 provider 연결, 설정 관리, 작업 취소, 파일 입출력 등 반복적으로 검증되어야 하는 기반 계층의 리스크를 낮추고, 연구의 초점을 고레버리지 질문 생성과 결정 기반 학습 자료 생성에 둘 수 있었다."
- "오픈소스 기반 사용은 LLM 출력의 의미론적 정확성을 보장하지는 않지만, 도구 인프라의 안정성과 구현 재현성에 대한 우려를 줄이는 보조 근거로 작동한다."

### 3.5 방어 가능한 범위

오픈소스 기반 사용으로 방어할 수 있는 부분:

- VS Code 확장 구조와 webview 기반 UI 패턴
- 모델 provider 설정과 호출 구조
- 개발자 도구에서 일반적으로 기대되는 작업 흐름
- 파일 생성, 상태 표시, 세션 관리 같은 도구 인프라
- 이미 커뮤니티에서 사용된 설계 패턴을 참고했다는 구현 타당성

오픈소스 기반 사용만으로 방어하면 안 되는 부분:

- AI가 생성한 코드가 항상 맞다는 주장
- 생성된 학습 자료가 항상 사실이라는 주장
- 질문이 항상 최적이라는 주장
- 특정 프로젝트의 요구사항을 완전히 이해했다는 주장

즉, 오픈소스 기반 사용은 "AI 판단의 신뢰성"이 아니라 "도구 기반 구조의 신뢰성"에 대한 방어 논리로 사용해야 한다.

### 3.6 발표 자료에 넣을 증거

발표 또는 논문 부록에는 다음 자료를 제시하면 좋다.

- Continue GitHub 링크: `https://github.com/continuedev/continue`
- star 수, fork 수, release 이력, contributor 수, 최근 commit 활동
- 라이선스와 재사용 가능 여부
- Debtcrasher가 새로 추가하거나 변경한 부분의 범위
- 기존 오픈소스 구조에서 유지한 부분과 Debtcrasher의 연구 기여 부분을 분리한 아키텍처 그림

권장 프레임:

```text
기반 계층: Continue의 VS Code extension/bootstrap/orchestration 구조
Debtcrasher 기여 계층: planning gate, high-leverage question filtering, decision log, tutorial generation
```

이렇게 나누면 교수님이 "이 도구 전체를 새로 만든 것이면 안정성이 의심된다"라고 질문했을 때, "기반 도구 계층은 검증된 공개 구조를 재사용했고, 연구는 그 위의 결정 학습 계층에 집중했다"라고 답할 수 있다.

## 4. 신뢰성 범위 정의

Debtcrasher가 보장하려는 신뢰성은 절대적 정답성이 아니다. 다음과 같이 범위를 좁혀야 방어가 쉽다.

### 4.1 보장하려는 것

- 사용자의 중요한 의사결정을 기록한다.
- AI가 임의로 결정하면 위험한 지점에서는 사용자에게 질문한다.
- 생성된 코드에 대해 가능한 경우 compile, build, typecheck, test 같은 자동 검증을 수행한다.
- 생성된 학습 자료는 실제 결정 로그와 최근 구현 맥락을 기반으로 작성한다.
- 실패한 검증 결과를 숨기지 않고 사용자에게 노출한다.
- 생성 파일 경로를 workspace 내부로 제한한다.
- 세션, 결정, 튜토리얼을 파일로 남겨 추적 가능하게 한다.

### 4.2 보장하지 않는 것

- 모든 코드가 완벽하다는 것
- 모든 요구사항 해석이 맞다는 것
- 모든 학습 자료가 교재 수준으로 완전하다는 것
- 개발자 검토 없이 바로 배포 가능한 결과를 만든다는 것
- AI가 프로젝트의 모든 숨은 의도를 이해한다는 것

논문에서는 이 한계를 명시하는 것이 오히려 신뢰성에 도움이 된다. Debtcrasher는 "무검토 자동 개발기"가 아니라 "개발자의 의사결정과 복습을 보조하는 개발자용 에이전트"이기 때문이다.

## 5. Friction-aware reliability 설계

개발자 도구에서 신뢰성 장치가 너무 무겁게 들어가면 도구 사용률이 떨어진다. 특히 코딩 중인 개발자는 빠른 피드백을 원하기 때문에, 모든 단계에서 긴 검증과 확인 절차를 요구하면 실제 사용성이 낮아질 수 있다.

따라서 Debtcrasher는 다음 원칙을 따른다.

- 저비용 검증은 기본 적용한다.
- 고비용 검증은 위험도가 높거나 사용자가 요청한 경우에만 적용한다.
- 사용자가 이미 판단해야 하는 자연스러운 지점에 질문을 배치한다.
- AI가 확신하지 못하는 내용을 숨기지 않고 가정 또는 추론으로 표시한다.
- 실패를 성공처럼 포장하지 않고 검증 실패 결과를 그대로 보여준다.

### 5.1 기본 모드와 엄격 모드

| 구분 | 기본 모드 | 엄격 모드 |
| --- | --- | --- |
| 대상 | 일반 개발 흐름 | 논문 실험, 데모, 중요한 변경 |
| 질문 수 | 최대 3개 | 최대 3개 유지 |
| 코드 검증 | 가능한 자동 명령만 실행 | test/build/typecheck 우선 실행 |
| 튜토리얼 검증 | 형식 검증, 근거 포함 | 근거 누락 검사, 2차 자기검토 |
| 장점 | 빠름, 사용 부담 낮음 | 발표와 실험에서 방어력 높음 |
| 단점 | 일부 오류 가능 | 시간 증가 |

이 구분을 두면 "왜 더 강한 검증을 하지 않았는가"라는 질문에 대해, 실제 사용자 흐름에서는 기본 모드를 사용하고 연구 평가나 중요한 상황에서는 엄격 모드를 사용할 수 있다고 답할 수 있다.

## 6. 고레버리지 질문 판단 기준

### 6.1 고레버리지 질문의 정의

고레버리지 질문이란 답변 하나가 이후 구현 방향, 아키텍처, 데이터 모델, 사용자 경험, 보안, 비용, 학습 자료의 해석에 큰 영향을 주는 질문이다.

반대로 저레버리지 질문은 코드베이스에서 쉽게 추론 가능하거나, 나중에 쉽게 수정 가능하거나, 사용자의 생산성을 끊을 만큼 중요하지 않은 질문이다.

Debtcrasher의 질문 전략은 "많이 묻기"가 아니라 "적게, 그러나 중요한 것만 묻기"다.

### 6.2 질문 생성 기준

다음 항목 중 여러 개에 해당하면 고레버리지 질문으로 판단한다.

| 기준 | 설명 | 예시 |
| --- | --- | --- |
| 되돌리기 비용 | 나중에 바꾸기 어렵거나 변경 범위가 커지는가 | DB schema, 상태 관리 구조 |
| 공개 계약 영향 | API, CLI, 파일 형식, 설정값처럼 외부 계약이 바뀌는가 | REST endpoint 이름, config key |
| 보안/권한 영향 | 인증, 권한, secret, 파일 접근과 관련되는가 | API key 저장 방식 |
| 데이터 손실 가능성 | 기존 데이터를 삭제, 이동, 덮어쓸 수 있는가 | migration, cleanup 로직 |
| 사용자 의도 모호성 | 여러 구현 방향이 모두 그럴듯한가 | "간단한 UI"의 의미 |
| 코드 근거 부족 | repo만 보고는 답을 추론하기 어려운가 | 교수님 요구사항, 실험 조건 |
| 파급 범위 | 여러 파일이나 모듈에 영향을 주는가 | shared service 변경 |
| 학습 가치 | 나중에 복습 자료로 남길 만한 설계 판단인가 | 왜 A 구조를 선택했는지 |

### 6.3 간단한 점수화 방식

각 후보 질문에 대해 아래 점수를 부여한다.

| 항목 | 점수 |
| --- | --- |
| 되돌리기 어렵다 | 2 |
| 보안, 데이터, 비용과 관련된다 | 2 |
| 외부 API나 사용자 경험에 영향을 준다 | 1 |
| 코드베이스에서 추론하기 어렵다 | 1 |
| 여러 선택지가 모두 타당하다 | 1 |
| 이후 학습 자료에 중요한 결정으로 남는다 | 1 |

운영 규칙:

- 총점 0-1점: 질문하지 않고 코드베이스 관례 또는 기본값으로 처리한다.
- 총점 2-3점: 다른 질문 수가 적을 때만 묻는다.
- 총점 4점 이상: 고레버리지 질문으로 사용자에게 묻는다.
- 보안, 데이터 손실, 유료 API 사용, public contract 변경은 점수와 무관하게 우선 질문한다.
- 최종 질문은 최대 3개로 제한한다.

### 6.4 질문 형식

질문은 열린 질문보다 선택지를 포함한 짧은 질문이 좋다.

좋은 예:

```text
이 기능의 저장 위치를 어디로 둘까요?
A. workspace 내부 markdown 파일
B. extension global storage
C. 별도 DB 또는 외부 저장소
```

나쁜 예:

```text
어떤 방식으로 구현할까요?
```

질문마다 다음 메타데이터를 함께 남기면 신뢰성 방어에 도움이 된다.

```json
{
  "question": "결정 로그를 어디에 저장할까요?",
  "leverage_score": 5,
  "reason": "저장 위치는 이후 세션 복원, 공유 가능성, 학습 자료 생성에 영향을 준다.",
  "default_if_skipped": "workspace 내부 DECISIONS.md",
  "risk_if_wrong": "나중에 저장 구조를 바꾸면 기존 로그 마이그레이션이 필요하다."
}
```

이 메타데이터는 사용자에게 모두 보여줄 필요는 없다. 하지만 내부 로그나 실험 분석에 남기면 "질문 선정이 임의적이지 않다"는 근거가 된다.

### 6.5 중복 질문 방지

신뢰성 문제는 질문 수를 늘리는 방식으로 해결하면 안 된다. 질문이 많아지면 사용자는 도구를 쓰지 않는다.

따라서 다음 장치를 둔다.

- 사용자가 답한 결정은 `DECISIONS.md`에 기록한다.
- 반복적으로 필요한 정책은 `AGENT.md`에 요약한다.
- 다음 planning 단계에서는 이전 결정과 충돌할 때만 다시 묻는다.
- 동일한 질문은 표현만 바꿔 다시 묻지 않는다.
- 질문하지 않고 기본값을 선택한 경우에도 "기본값으로 처리한 가정"을 기록한다.

## 7. 학습 자료 생성 신뢰성 대책

### 7.1 기본 문제

학습 자료 생성에서 교수님이 우려할 수 있는 부분은 다음과 같다.

- AI가 실제로 존재하지 않는 이유를 꾸며낼 수 있다.
- 코드 변경과 무관한 일반론을 학습 자료처럼 만들 수 있다.
- 사용자가 선택하지 않은 의사결정을 선택한 것처럼 설명할 수 있다.
- 실패한 구현이나 검증 결과를 성공 사례처럼 포장할 수 있다.
- 여러 step을 묶을 때 시간 순서나 인과관계를 왜곡할 수 있다.

Debtcrasher는 이 문제를 "완벽한 사실 검증"으로 해결하기보다, 생성 자료의 근거 범위를 제한하는 방식으로 줄인다.

### 7.2 근거 기반 생성 원칙

튜토리얼 생성 시 사용할 수 있는 근거를 명시적으로 제한한다.

허용 근거:

- 사용자가 planning 단계에서 선택한 답변
- `DECISIONS.md`에 저장된 step 로그
- `AGENT.md`에 저장된 프로젝트 정책과 최근 맥락
- 최근 구현 결과 요약
- 자동 검증 명령과 결과
- 실제 생성 또는 수정된 파일 경로

주의가 필요한 근거:

- 모델의 일반 지식
- 코드베이스에 없는 best practice
- 사용자가 말하지 않은 요구사항
- 실패 여부가 확인되지 않은 구현 결과

논문 표현:

> 학습 자료 생성은 자유 생성 방식이 아니라 decision-grounded generation 방식으로 제한한다. 생성 모델은 사용자의 결정 로그, 구현 요약, 검증 결과를 입력 근거로 사용하며, 근거가 없는 내용은 일반화된 설명이 아니라 가정 또는 추가 확인 항목으로 표시하도록 설계한다.

### 7.3 근거 라벨링

생성된 학습 자료에는 다음 라벨 중 하나를 붙이는 방식을 사용할 수 있다.

| 라벨 | 의미 |
| --- | --- |
| 사용자 결정 | planning 단계에서 사용자가 직접 선택한 내용 |
| 코드 근거 | 실제 파일 또는 구현 결과에서 확인되는 내용 |
| 검증 결과 | build, test, typecheck 등 명령 결과 |
| AI 추론 | 근거는 있으나 직접 확인된 사실은 아닌 해석 |
| 확인 필요 | 근거가 부족하거나 충돌 가능성이 있는 내용 |

예시:

```markdown
## 왜 이 결정을 했는가

- [사용자 결정] 튜토리얼은 workspace 내부 `.ai-tutorials`에 저장하기로 했다.
- [코드 근거] Step View는 `DECISIONS.md`에서 선택된 step을 읽어 문서를 생성한다.
- [AI 추론] 이 구조는 학습 자료를 프로젝트와 함께 보관하기 위한 선택으로 볼 수 있다.
- [확인 필요] 팀 단위 공유를 고려한다면 저장 위치 정책을 별도로 정해야 한다.
```

이 방식은 학습 자료 전체를 완전히 검증하지 않아도, 독자가 어떤 문장이 사실이고 어떤 문장이 해석인지 구분할 수 있게 한다.

### 7.4 튜토리얼 필수 구조

튜토리얼 생성 결과에는 최소한 다음 섹션이 있어야 한다.

```markdown
# 제목

## 선택한 결정
## 당시 맥락
## 선택하지 않은 대안
## 이 결정이 구현에 준 영향
## 나중에 다시 확인할 점
## 관련 파일 또는 로그
```

검증 규칙:

- 필수 섹션이 없으면 저장하지 않는다.
- 선택한 step과 관련 없는 일반론만 있으면 저장하지 않는다.
- 관련 파일 또는 로그가 하나도 없으면 경고를 표시한다.
- 검증 실패가 있었는데 성공처럼 작성하면 경고를 표시한다.
- "항상", "완벽히", "보장한다" 같은 강한 표현은 피한다.

### 7.5 낮은 비용으로 가능한 검증

당장 구현 부담이 낮은 대책:

- 튜토리얼 markdown에 필수 heading이 있는지 검사한다.
- 선택된 step id 또는 step title이 본문에 포함되어 있는지 검사한다.
- 관련 파일 경로가 하나 이상 포함되어 있는지 검사한다.
- 검증 실패가 있는 경우 "검증 결과" 섹션을 반드시 포함시킨다.
- 모델 출력이 너무 짧으면 저장하지 않는다.
- "근거 없음", "추정", "확인 필요" 같은 표현을 허용해 모델이 억지로 단정하지 않게 한다.

조금 더 강한 대책:

- 생성 후 2차 모델 호출로 "근거 없는 주장 찾기"만 수행한다.
- tutorial strict mode에서만 2차 검토를 실행한다.
- 각 문단 옆에 근거 source id를 붙인다.
- 사람이 수정할 수 있도록 생성 직후 editor에서 바로 열어준다.

## 8. 코드 생성 신뢰성 대책

Debtcrasher의 코드 생성 신뢰성은 완벽한 정답 보장이 아니라 "개발자가 검토 가능한 후보 결과를 만들고, 명백한 실패를 빠르게 드러내는 것"에 가깝다.

기본 대책:

- AI 응답은 JSON schema로 강제한다.
- 파일 경로는 workspace 내부로 제한한다.
- 생성 파일과 수정 파일을 구분한다.
- 가능한 경우 자동 검증 명령을 실행한다.
- 검증 실패 시 1회 자동 수정 후 재검증한다.
- 그래도 실패하면 실패 로그를 사용자에게 보여준다.

논문 방어 포인트:

> Debtcrasher는 생성 결과를 무조건 성공으로 간주하지 않는다. 가능한 자동 검증 명령을 탐지해 실행하고, 실패한 경우 한 차례 repair loop를 수행한다. 이 과정은 완전한 정답성을 보장하기 위한 것이 아니라, 개발자가 즉시 확인 가능한 실패를 줄이고 생성 결과의 상태를 투명하게 드러내기 위한 최소 검증 장치다.

## 9. 교수님 예상 질문과 답변

### Q1. 기존 오픈소스를 썼다는 것이 왜 신뢰성 근거가 되는가?

기존 오픈소스 사용은 AI 출력의 정확성을 보장하는 근거가 아니다. 다만 VS Code 확장 구조, 모델 provider 연결, webview 통신, 작업 상태 관리처럼 도구 기반 계층을 처음부터 새로 만들 때 생길 수 있는 엔지니어링 리스크를 줄이는 근거다. Debtcrasher의 연구 기여는 그 기반 위에 planning gate, 결정 로그, 학습 자료 생성을 결합한 부분에 있다.

### Q2. AI가 잘못된 질문을 하면 어떻게 하는가?

질문은 임의로 생성하지 않고 고레버리지 기준에 따라 제한한다. 되돌리기 어렵거나, 보안/데이터/API에 영향을 주거나, 여러 선택지가 모두 타당한 경우에만 질문한다. 또한 질문 수를 최대 3개로 제한하고, 사용자의 답변은 decision log에 기록해 반복 질문을 줄인다.

### Q3. 학습 자료가 환각을 포함할 수 있지 않은가?

가능하다. 따라서 Debtcrasher는 학습 자료를 일반 지식 기반으로 자유 생성하지 않고, 사용자가 선택한 결정 로그, 구현 요약, 검증 결과, 관련 파일 경로를 근거로 제한한다. 또한 근거가 없는 내용은 단정하지 않고 추론 또는 확인 필요 항목으로 표시하도록 한다.

### Q4. 왜 더 강한 검증을 하지 않는가?

Debtcrasher의 대상은 개발자이며, 도구의 목적은 생산성 향상과 의사결정 학습 보조다. 모든 생성 결과에 긴 검증 절차를 붙이면 사용 시간이 늘어나고 실제 사용률이 낮아질 수 있다. 따라서 기본 모드에서는 저비용 검증만 수행하고, 논문 실험이나 중요한 변경에서는 strict mode로 더 강한 검증을 수행할 수 있게 설계한다.

### Q5. 결국 사용자가 검토해야 한다면 AI 도구의 의미가 약한 것 아닌가?

개발 도구에서 사용자 검토는 약점이 아니라 전제다. Debtcrasher의 목표는 개발자를 대체하는 것이 아니라, 구현 전에 중요한 결정을 드러내고, 그 결정을 나중에 학습 가능한 자료로 바꾸는 것이다. 사용자는 여전히 최종 판단권을 갖지만, 의사결정 누락과 복습 비용이 줄어든다.

## 10. 논문용 표현 예시

### 10.1 신뢰성 설계 설명

```text
본 연구는 생성형 AI의 출력을 무조건 신뢰하는 완전 자동 개발 시스템을 목표로 하지 않는다.
대상 사용자를 개발자로 한정하고, 개발자가 코드와 실행 결과를 검토할 수 있다는 전제 위에서
생산성을 해치지 않는 수준의 저마찰 신뢰성 장치를 설계하였다.
```

### 10.2 오픈소스 기반 설명

```text
Debtcrasher는 AI 개발 에이전트 래퍼의 기반 구조를 처음부터 구현하지 않고,
이미 공개적으로 사용되고 유지보수되는 오픈소스 개발 에이전트 구조를 참고 및 변형하였다.
이를 통해 모델 provider 연결, VS Code 확장 생명주기, webview 기반 상호작용 등
도구 인프라 계층의 구현 리스크를 낮추고, 연구 기여를 고레버리지 질문 생성과
결정 기반 학습 자료 생성에 집중하였다.
```

### 10.3 트레이드오프 설명

```text
개발자용 AI 도구에서 신뢰성 장치는 정확성뿐 아니라 사용 지속성에도 영향을 준다.
과도한 확인 절차와 긴 검증 시간은 사용자의 흐름을 방해하고 도구 채택률을 낮출 수 있다.
따라서 Debtcrasher는 모든 단계에서 동일한 수준의 검증을 적용하기보다,
되돌리기 어렵거나 파급 효과가 큰 결정 지점에만 사용자 확인을 요구하는
위험 기반 검증 전략을 채택하였다.
```

### 10.4 학습 자료 생성 설명

```text
학습 자료 생성은 자유로운 설명 생성이 아니라 결정 로그에 근거한 문서화 과정으로 설계하였다.
생성 모델은 사용자의 선택, 결정 시점의 맥락, 구현 결과, 자동 검증 결과를 입력으로 받아
튜토리얼을 생성하며, 확인되지 않은 내용은 단정적 설명이 아니라 추가 확인 항목으로 표시한다.
```

## 11. 실험과 평가 아이디어

신뢰성 우려를 논문에서 방어하려면 기능 설명만으로는 부족하다. 최소한 다음 평가 중 일부를 넣으면 좋다.

중요한 범위 경계는 분명하다. 사용자 피드백 수집과 질문 유용성 로그 저장은 현재 구현 범위에 넣지 않고, 배포 이후의 사용자 실험이나 설문을 통한 향후 평가 계획으로만 둔다.

### 11.1 고레버리지 질문 평가

평가 방법:

- 여러 개발 과제를 준비한다.
- Debtcrasher가 생성한 질문을 수집한다.
- 개발 경험이 있는 평가자가 각 질문을 고레버리지/저레버리지로 라벨링한다.
- 질문 precision, 불필요 질문 수, 누락된 중요 질문 수를 측정한다.

가능한 지표:

- high-leverage question precision
- task당 평균 질문 수
- 사용자가 불필요하다고 느낀 질문 비율
- 질문 답변 이후 구현 수정 감소 여부

### 11.2 학습 자료 신뢰성 평가

평가 방법:

- 생성된 튜토리얼의 문장을 근거 유형별로 분류한다.
- 각 문장이 `DECISIONS.md`, `AGENT.md`, 코드 diff, 검증 결과 중 하나로 뒷받침되는지 확인한다.
- 근거 없는 주장 비율과 명백한 오류 비율을 측정한다.

가능한 지표:

- source-supported statement ratio
- unsupported claim ratio
- contradiction count
- developer usefulness rating
- review 후 수정 필요 문장 수

### 11.3 생산성 평가

평가 방법:

- 일반 AI 코딩 에이전트만 사용한 경우와 Debtcrasher를 사용한 경우를 비교한다.
- 구현 시간, 질문 수, 생성된 학습 자료의 품질, 나중에 결정 이유를 회상하는 능력을 측정한다.

가능한 지표:

- task completion time
- post-task recall accuracy
- number of missing design decisions
- perceived interruption cost
- perceived usefulness

이 평가를 통해 "검증을 많이 해서 신뢰성을 높였다"가 아니라 "생산성을 유지하면서 중요한 실패 지점을 줄였다"는 주장을 만들 수 있다.

## 12. 당장 구현하면 좋은 최소 대책

가장 적은 구현 비용으로 논문 방어력을 높이는 항목은 다음과 같다.

1. planning 응답에 `leverage_score`, `reason`, `default_if_skipped` 필드를 추가한다.
2. 질문 수는 계속 최대 3개로 제한하되, 고레버리지 기준을 prompt와 문서에 명시한다.
3. 튜토리얼 markdown에 필수 섹션 validator를 둔다.
4. 튜토리얼에 "관련 결정 로그"와 "관련 파일" 섹션을 반드시 넣는다.
5. 검증 실패가 있었던 경우 튜토리얼에 "검증 결과" 또는 "확인 필요" 섹션을 강제한다.
6. 발표 자료에 오픈소스 기반 계층과 Debtcrasher 고유 기여 계층을 분리한 그림을 넣는다.
7. strict mode를 옵션으로 두고, 논문 실험에서는 strict mode를 사용했다고 설명한다.

## 13. 최종 포지셔닝

Debtcrasher의 신뢰성 포지션은 다음 문장으로 정리할 수 있다.

```text
Debtcrasher는 AI의 모든 판단을 신뢰하는 도구가 아니라,
개발자가 놓치기 쉬운 고레버리지 결정을 드러내고,
그 결정과 구현 맥락을 추적 가능한 학습 자료로 바꾸는 개발자 중심 보조 도구다.
신뢰성은 완전 자동 검증이 아니라 오픈소스 기반 구조, 위험 기반 질문 필터링,
자동 검증 결과 노출, 결정 로그 기반 문서화를 통해 확보한다.
```
