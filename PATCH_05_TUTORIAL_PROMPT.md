# PATCH 05 - 튜토리얼 프롬프트 재구성

## 무엇이 문제였는가

- 기존 `STEP_SYSTEM_PROMPT`는 일부 한글이 깨져 있었고, 출력 구조도 Debtcrasher가 목표로 하는 학습 문서 수준까지 강제하지 못했습니다.
- 그 결과 문서가 단순 회고나 얕은 요약에 머물 가능성이 컸고, 선택 간 연결과 오판 신호, 재사용 가능한 판단 규칙이 충분히 드러나지 않았습니다.

## 새 STEP_SYSTEM_PROMPT 구조

### 문서 상단
- `# [프로젝트명] — 판단 기록`
- `## 프로젝트 맥락`
  - 프로젝트 목표와 제약을 2~3문장으로 정리

### 각 판단 step마다 강제되는 섹션
- `### [판단 제목]`
- `결정한 것`
- `왜 필요했나`
- `선택지 비교표`
- `이 프로젝트에서 선택한 이유`
- `이 선택이 이후 결정에 미친 영향`
- `이 판단이 틀렸을 때 나타날 신호`
- `다음에 비슷한 상황이 오면`

### 여러 step을 함께 생성할 때 문서 마지막에 추가되는 섹션
- `## 판단들의 연결 구조`
- `## 내 판단 패턴 분석`

## 프롬프트에 추가된 핵심 지시

- 출력 전체를 한국어로 작성
- 비교표만 markdown 표 사용
- 본문 섹션은 bullet 대신 짧은 문단형 prose로 작성
- step 하나당 600~900자 안팎으로 작성
- `이 선택이 이후 결정에 미친 영향`은 반드시 다른 판단과 연결해서 설명
- `이 판단이 틀렸을 때 나타날 신호`는 관찰 가능한 증상으로 구체화
- `다음에 비슷한 상황이 오면`은 다음 프로젝트에 재사용할 수 있는 판단 규칙으로 작성
- 전체 톤은 교과서가 아니라 시니어 개발자의 복기 문체로 유지

## 변경된 PATTERN_GROUPS

### 기존 그룹 정리
- `simple-fast`
  - 설명: `간단함과 빠른 전달을 우선하는 경향`
  - 키워드 보강: `간단`, `빠른`, `최소`, `프로토타입`, `가볍게`, `빨리`
- `local-static`
  - 설명: `로컬 또는 정적 구조를 우선하는 경향`
  - 키워드 보강: `정적`, `로컬`, `워크스페이스`, `파일 기반`, `백엔드 없음`, `오프라인`
- `extensible`
  - 설명: `확장성과 유연성을 우선하는 경향`
  - 키워드 보강: `확장`, `유연`, `모듈`, `재사용`, `스케일`, `플러그인`
- `explicit-control`
  - 설명: `명시적인 사용자 통제를 우선하는 경향`
  - 키워드 보강: `명시`, `직접`, `수동`, `사용자 선택`, `판단`, `승인`

### 새로 추가한 그룹
- `performance-first`
  - 설명: `성능과 응답 속도를 우선하는 경향`
  - 키워드: `성능`, `속도`, `응답 속도`, `지연`, `최적화`, `효율`
- `security-first`
  - 설명: `보안과 안전한 경계 설정을 우선하는 경향`
  - 키워드: `보안`, `인증`, `인가`, `비밀키`, `개인정보`, `암호화`
- `ux-first`
  - 설명: `사용자 경험과 사용성을 우선하는 경향`
  - 키워드: `사용자 경험`, `사용성`, `온보딩`, `명확성`, `접근성`, `인터랙션`

## 톤 차이를 보여주는 예시 발췌

이 판단이 필요했던 이유는 단순히 구현 취향의 문제가 아니라, 이후 파일 구조와 검증 방식이 모두 달라지는 갈림길이었기 때문이다. 당시에는 빠른 프로토타입이 우선이었기 때문에 확장 가능한 구조보다 즉시 실행 가능한 구조가 더 높은 가치를 가졌다. 이 선택 덕분에 초기 구현 속도는 빨라졌지만, 이후 상태 공유가 늘어날 때 구조 재정리가 필요해지는 제약도 함께 생겼다. 다음에 비슷한 상황이 오면, 데모 속도가 최우선일 때만 단순 구조를 먼저 택하고 상태 결합이 빨리 커질 조짐이 보이면 초기에 한 단계 더 구조화를 고려하는 편이 낫다.
